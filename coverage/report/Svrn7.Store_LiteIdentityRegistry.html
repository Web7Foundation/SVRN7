 239/// <summary>
 240/// LiteDB-backed nonce store for transfer replay protection (Step 3).
 241///
 242/// Design:
 243///   - The Nonces collection has a unique index on Nonce and a non-unique index on ExpiresAt.
 244///   - IsReplayAsync first sweeps expired entries (ExpiresAt &lt; UtcNow), then attempts to
 245///     insert the new nonce. A duplicate-key exception on insert means the nonce is a replay.
 246///   - The sweep runs on every call — this is acceptable because nonce checks are already
 247///     on the hot path and the expired set is expected to be small (24-hour window × call rate).
 248///   - Thread safety: LiteDB write operations are serialised internally; no additional
 249///     locking is required for single-process deployments.
 250///   - Multi-process deployments: share a single svrn7.db file path; LiteDB handles
 251///     file-level locking. Only one writer process at a time is supported.
